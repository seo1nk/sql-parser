# UI 設計（Liam ERD を参考にしたデータフロー可視化）

参考: [liam-hq/liam](https://github.com/liam-hq/liam)（/Users/seoink/Documents/liam）

## 目的・利用用途

SELECT 文を「結果の表がどう作られるか」の **左→右のデータフローグラフ** として描く。

1. **長文 SQL の読解** … WITH / サブクエリ / JOIN が入り組んだクエリの構造を一目で把握する
2. **SQL チューニング** … データフローの並びが **論理実行順**（FROM → JOIN → WHERE →
   GROUP BY → HAVING → SELECT → ORDER BY → LIMIT）と一致するため、
   「どの段階で行数が膨らむか / 絞られるか」を考える土台になる。
   将来的には各エッジに推定行数や EXPLAIN 結果を重ねる拡張も視野に入れる

## 描画のルール

### ノードとステップ

- **WITH / FROM / サブクエリ** … データの供給源（テーブルノード・CTE ノード）
- **JOIN** … **JOIN ノードは作らない**。結合される 2 つの集合が **合流して
  1 つの結合済みテーブルノードになる**（例: `adults ⋈ orders`）。
  結合キーは **それぞれの矢印（エッジ）上にラベルとして描く**
  （adults 側の矢印に `a.id`、orders 側の矢印に `o.user_id`）。
  JOIN 種別（INNER / LEFT …）は合流先ノードのヘッダーに表示する
- **WHERE / HAVING** … フィルタノード（条件式を表示）
- **GROUP BY** … グループ化ノード
- **SELECT** … 射影ノード（最終段は結果テーブルとして列を表示）

各ノードには論理実行順のステップ番号バッジ（① WITH ② FROM ③ JOIN ④ WHERE
⑤ GROUP BY ⑥ SELECT …）を付け、下部の実行順タイムラインと対応させる。

### カラム表示は「SQL に現れた事実」のみ

- テーブルノードに表示する列は、**SQL 中に実際に登場した列だけ**にする。
  スキーマ情報や推測での補完はしない（パーサーが返せるのは事実のみ、という原則）
- SQL に現れていない列が存在しうるテーブルには、列リストの末尾に `…` を描画して
  「他にも列があるかもしれない」ことを示す
- `SELECT *` のように全列が流れることだけ分かる場合も `*` / `…` で表現する

### 列の系譜（lineage）の色分け

| 表示 | 意味 |
| --- | --- |
| 緑（アクセント） | 最終結果に値が届く列（`count(o.id)` → `order_count` のような集計経由を含む） |
| 白 | 結合キー・WHERE / GROUP BY などの条件でのみ使われる列 |
| `…`（減光） | SQL に現れていない未知の列の存在を示すプレースホルダ |

## 技術スタック

| 項目 | 採用 | 備考 |
| --- | --- | --- |
| ビルド / FW | **Vite + React + TypeScript** | |
| キャンバス | **@xyflow/react (React Flow) 12** | Liam と同じ |
| 自動レイアウト | **elkjs**（`elk.algorithm: layered`）| 層状レイアウトはデータフローと相性が良い |
| UI コンポーネント | **Base UI（@base-ui-components/react）** | Radix の代わり。ヘッドレスなので Tailwind と組み合わせる |
| スタイリング | **Tailwind CSS** | デザイントークンは Tailwind の theme 拡張として定義 |
| バリデーション | valibot | AST(JSON) の受け取りに採用検討 |
| パーサー連携 | **wasm-bindgen + wasm-pack**（Rust→WASM）| Liam の pg-query-emscripten 相当。型定義が自動生成される |

### アーキテクチャパターン（Liam erd-core の構成を踏襲）

```
frontend/src/features/flow/
├── utils/       … 純粋関数: AST(Step[]) → ノード/エッジ変換、ハイライト計算、ELKレイアウト
├── components/  … カスタムノード（TableNode, FilterNode, GroupNode, ProjectNode…）とカスタムエッジ
└── hooks/       … 初期レイアウト（ノード実測後に ELK 実行 → fitView）、選択状態
```

- 「AST → グラフ」「グラフ → レイアウト」「グラフ → ハイライト」を純粋関数として分離
  （Liam の `convertSchemaToNodes` / `getElkLayout` / `highlightNodesAndEdges` に対応）
- JOIN の合流は「AST → グラフ」変換で行う: `join` ステップは
  「左右の入力ノード → 結合済みテーブルノードへの 2 本のエッジ（キーラベル付き）」に展開する

### デザイントークン（kawaii × 見やすい）

Liam からは**構成・インタラクションパターンのみ**借り、ビジュアルは独自路線。
ダークではなく、ミルキーな明るい背景にパステル + はっきりしたインク色のペアで
「ポップでかわいいが、読みやすい」を両立する。Tailwind の `@theme` として定義。

| トークン | 値 | 用途 |
| --- | --- | --- |
| キャンバス背景 | `#fff7f9`（さくらミルク）+ ドット `#f3d9e1` | |
| ノード背景 | `#ffffff`、角丸 16px、ボーダー 2px `#f0e0e8` | |
| 文字色（ink） | `#4a3b44` / muted `#8a7a84` / dim `#b9abb3` | 黒より柔らかい暖色系グレー |
| アクセント | `#ff7dae`（ピンク）/ 文字用 `#d6467e` / 淡 `#ffd3e4` | 選択・ホバー・結果に届く列 |
| ハイライト | `border: #ff7dae; box-shadow: 0 6px 20px rgba(255,125,174,.35)` | ピンクのやわらかい影 |
| フォント | M PLUS Rounded 1c / M PLUS 1 Code | 丸ゴシックで kawaii、日本語対応 |

ステップ種別カラーは「淡い背景（soft）× 濃い文字（ink）」のパステルペア。
バッジは soft 背景に ink 文字の丸チップで、視認性を確保する:

| ステップ | soft | ink |
| --- | --- | --- |
| scan / cte（供給源） | `#bdf0e3` | `#0e8c74`（ミント） |
| 結合済みテーブル | `#e5d8ff` | `#7c4dd8`（ラベンダー） |
| filter（WHERE / HAVING） | `#ffe3bc` | `#c77414`（アプリコット） |
| group | `#cde5ff` | `#2b7cd9`（ソーダ） |
| project（SELECT / 結果） | `#ffd3e4` | `#d6467e`（ピンク） |

### データフロー演出

- エッジは矢印付きベジェ曲線（`#dbc7d2`、2px）。ハイライト時（ノードホバーで
  **上流経路全体**が対象）はピンク + やわらかい影 + パーティクル
  （Liam の RelationshipEdge の `<animateMotion>` パターン）
- `prefers-reduced-motion` ではパーティクルを止める

## 画面構成

```
┌─────────────────────────────────────────────────────┐
│ ヘッダー                                              │
├───────────────┬─────────────────────────────────────┤
│ SQL エディタ    │  フローキャンバス（React Flow）         │
│ (左ペイン)     │  - パン/ズーム、ドット背景               │
│ ステップ凡例    │  - ノードホバーで上流経路をハイライト     │
│ パース結果/    │                                     │
│ エラー表示     ├─────────────────────────────────────┤
│               │ 実行順タイムライン（①WITH→…→⑥SELECT）   │
│               │ ホバー/クリックで該当ノードをハイライト    │
└───────────────┴─────────────────────────────────────┘
```

## 実装ステップ

1. **モックアップ**（HTML 一枚）… ビジュアル言語の確認 ← 済・フィードバック反映済み
2. **フロントエンド雛形** … `frontend/` に Vite + React + TS + Tailwind + Base UI +
   @xyflow/react + elkjs。モック `Step[]`（[06_api_design.md](./06_api_design.md)）でキャンバスを動かす
3. **カスタムノード/エッジ** … 種別ごとのノード、キーラベル付きエッジ、パーティクル
4. **Rust 側の追いつき** … フェーズ3（AST）→ フェーズ4（wasm-pack）完成後、
   モックデータを `parse()` / `explain()` の実出力に差し替える
5. **チューニング支援（将来）** … 各エッジへの行数・コスト表示、EXPLAIN 連携
